# Artifacts Guide — 문서 유형별 산출물 가이드

> 기존 agentic team 의 페르소나별 산출물 포맷을 **문서 유형 기준**으로 재편.
> 페르소나 라벨은 없어졌지만, **산출물의 구조와 품질 기준은 그대로** 유지.

## 개요

에이전트가 작업 중 생성하는 모든 문서는 **목적 기반** 분류로 `docs/` 하위에 저장된다.
"누가 작성했는가" 가 아니라 "무엇을 위한 문서인가" 가 기준이다.

| 용도 | 경로 | 언제 작성 |
|---|---|---|
| 외부 자료 수집/정리 | `docs/sources/` | 구글시트·CSV·논문 등 외부 자료를 팀 기준으로 구조화 |
| 제품 요구사항 | `docs/prd/` | 기능 기획 확정 전 명세 |
| 기획·리뷰·대안 분석 | `docs/reviews/` | PRD/아키텍처 리뷰, trade-off 비교 |
| UI/UX 설계 | `docs/design/` | 화면 구조·컴포넌트·인터랙션 설계 |
| 데이터 스펙 | `docs/data/` | ERD·스키마·거버넌스·분석 요구사항 |
| 보안 리뷰 | `docs/security/` | 위협 모델·취약점·승인 판단 |
| QA·배포 체크 | `docs/qa/` | 테스트 결과·배포 체크리스트 |
| 도메인 온톨로지 | `docs/domain/` | 3PL Object/Link/Action (팔란티어식) |
| 에이전트 질문 | `docs/questions/` | 사용자 응답 대기 중인 불명확점 |
| 세션 로그·레슨런 | `hyperagents/logs/` | **NEW**: 기존 `docs/chronicle/` 는 과거 기록으로 보존 |

## 1. External Sources (`docs/sources/`)

**목적**: 외부 문서 (Google Sheets, Notion, 웹, 논문) 를 팀이 소비할 수 있는 구조화된 마크다운으로 변환.

**템플릿**:
```markdown
# [문서 제목]

## 원본 정보
- 출처: [URL 또는 파일명]
- 수집일: [YYYY-MM-DD]
- 데이터 범위: [설명]

## 요약
[이 문서가 무엇인지, 핵심 내용 3-5줄]

## 데이터 구조
| 필드 | 타입 | 설명 | 비고 |

## 주요 발견사항
- ...

## 후속 문서를 위한 노트
- [PRD 에 반영할 점]
- [Dev 가 알아야 할 점]
```

**금지**: 데이터 임의 해석 (사실만 기록), 기획 판단.

---

## 2. PRD (`docs/prd/`)

**목적**: 사용자 요구사항을 구체적 기능 명세로 발전.

**템플릿**:
```markdown
# PRD: [기능명]

## 배경 및 목적
## 목표 (측정 가능)
## 유저 스토리
- [ ] US-001: As a ..., I want ..., so that ...

## 기능 명세
### P0 (Must Have)
| ID | 기능 | 설명 | 수용 기준 |

### P1 / P2

## 비기능 요구사항
- 성능 / 보안 / 확장성

## 범위 제외 (Out of Scope)
## 의존성
## 일정 (선택)
```

**품질 체크**: 유저 페르소나 체크리스트 — 비회원·신규·노년 등 빠진 페르소나 없는지.

---

## 3. Reviews (`docs/reviews/`)

**목적**: PRD 리뷰, 기술 선택 trade-off, Pre-flight/Checkpoint Review 산출물.

**템플릿 (PRD 리뷰)**:
```markdown
# Review: [대상 문서]

## 종합 평가
## 주요 피드백
### 보완 필요
- [ ] [항목]: [이유] → [제안]

### 리스크
| 리스크 | 영향도 | 발생 확률 | 완화 방안 |

### 대안 분석
| 옵션 | 장점 | 단점 | 추천 |

## 수정 요청
- ...
```

**템플릿 (Pre-flight / Checkpoint)**: `hyperagents/review_protocols.md` 참조.

## 4. Design (`docs/design/`)

**목적**: 확정 PRD → UI/UX 설계.

**템플릿**:
```markdown
# Design: [기능명]

## 화면 목록
| 화면 ID | 화면명 | 진입 경로 | 설명 |

## 유저 플로우
[시작] → [화면A] → [분기] → [화면B/C]

## 화면 상세
### S-001: [이름]
- 레이아웃
- 컴포넌트
- 인터랙션

## 컴포넌트 설계
| 컴포넌트 | Props | 상태 | 비고 |

## 상태 관리
[어디서 관리]
```

**품질 체크**: 반응형·접근성·로딩/에러 상태 정의.

## 5. Data (`docs/data/`)

**목적**: 데이터 거버넌스·모델링·파이프라인·분석 스펙.

**템플릿**:
```markdown
# Data Spec: [도메인]

## 데이터 카탈로그
| 엔티티 | 설명 | 소유자 | 분류 | 보존 |

## 데이터 모델 (ERD·스키마)
## 데이터 흐름
[원본] → [수집] → [가공] → [저장] → [서빙]

## 품질 규칙
| ID | 대상 | 검증 | 심각도 | 처리 |

## 접근 제어
| 역할 | 접근 | 허용 작업 |

## 분석 지표
| 지표 | 정의 | 산출 | 갱신 주기 |

## 구현 가이드
- 인덱싱 / 쿼리 최적화 / 마이그레이션
## 보안 노트
- 민감 데이터 / 암호화 / 감사 로그
```

## 6. Security (`docs/security/`)

**목적**: STRIDE 위협 모델·의존성 점검·프롬프트 인젝션·최종 승인 판단.

**템플릿**:
```markdown
# Security Review: [기능]

## 위협 모델 (STRIDE)
| 위협 | 대상 | 시나리오 | 영향도 | 완화 |

## 의존성 점검 (CVE)
## 프롬프트 인젝션 방어 (해당 시)
## 민감정보 노출 점검
## 코드 보안 (OWASP Top 10)
## 데이터 보안 (암호화·접근제어·감사)

## 발견된 이슈
| ID | 심각도 | 카테고리 | 설명 | 수정 방안 |

## 승인 판단
- 상태: [승인 / 조건부 / 거부]
- 근거 + 잔여 리스크
```

**프로세스**: Pre-flight (PRD 단계 보안 요구사항) → 구현 중 (OWASP 체크) → 배포 전 (최종 승인).

## 7. QA (`docs/qa/`)

**목적**: 테스트·배포 체크·품질 게이트.

**템플릿**:
```markdown
# QA Report: [기능]

## 테스트 결과
| 구분 | 전체 | 통과 | 실패 | 커버리지 |
| Unit / Integration / E2E / Security |

## 발견된 이슈
| ID | 심각도 | 설명 | 재현 | 수정 담당 |

## 배포 체크리스트
- [ ] 환경 변수 / DB 마이그레이션 / 롤백 / 모니터링 / 보안 승인

## 종합 판단
[배포 가능 여부 + 근거]
```

## 8. Domain (`docs/domain/`)

**목적**: 3PL 영업·운영 도메인 지식을 **팔란티어식 Object/Link/Action 온톨로지**로 축적.

**구조**:
```
docs/domain/
├── ontology.md            # 전체 뷰
├── objects/               # Object Type 상세
├── links/                 # 관계 정의
├── actions/               # 비즈니스 프로세스
├── glossary.md            # 용어집
├── rules.md               # 업계 규정
└── changelog.md           # 변경 이력
```

**Object 포맷**: 정의 / 속성 (확실도·출처) / 관계 / 관련 Action / 비즈니스 규칙 / 인사이트
**Action 포맷**: 정의 / 트리거 / 입력 객체 / 프로세스 플로우 / 상태 변화 / 예외 / 비즈니스 규칙

**확실도**:
- **높음**: 사용자 직접 전달/확인
- **중간**: 공식 문서에서 추출 (사용자 미확인)
- **낮음**: 추론 (표기 + 사용자 확인 요청)

## 9. Questions (`docs/questions/`)

**목적**: 에이전트가 작업 중 발견한 **사용자 응답 필요 사항** 누적.

**템플릿**:
```markdown
# Q: [짧은 제목]

## 맥락
[왜 이 질문이 필요한가]

## 선택지 (있을 경우)
- A: ...
- B: ...

## 영향 범위
[어떤 작업이 이 답변에 의존하는가]

## 상태
- [ ] 답변 대기 / [ ] 해결
```

## 10. Session Logs (`hyperagents/logs/`)

**목적**: 세션별 작업 로그·레슨런·하네스 개선 후보. **Chronicle Agent 역할 흡수**.

**생성**: `bash hyperagents/scripts/new-log.sh "<topic>"` (템플릿 자동)

**구조**:
- 컨텍스트 / 관찰 / 판단 / 남은 숙제 / 하네스 개선 후보

**Chronicle 과의 관계**: `docs/chronicle/` 은 과거 기록 아카이브로 보존. 앞으로 모든 세션 로그는 `hyperagents/logs/` 에.

---

## 교차 참조 규칙

- PRD 작성 시 `docs/domain/` 으로 도메인 용어 일관성 검증
- Design 시작 전 확정 PRD (`docs/prd/`) 필독
- Dev 구현 시 PRD + Design 둘 다 참조
- Security 는 PRD 확정 전 Pre-flight + 구현 중 + 배포 전 3단계
- QA 는 PRD 의 수용 기준 + Security 의 테스트 시나리오 기반

## 금지 사항

- 페르소나 라벨 (`@PRD`, `@Dev` 등) 사용 금지 — 혼란 유발
- 산출물 경로 임의 변경 금지 — 팀 표준 유지
- 한 문서에 여러 유형 혼재 금지 (예: PRD 안에 Design 포함) → 각 경로로 분리
