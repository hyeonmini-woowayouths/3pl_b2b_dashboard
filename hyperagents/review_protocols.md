# Review Protocols — 메타인지 점검 3단계

> 기존 Sage Agent 의 역할을 **프로토콜**로 흡수. 에이전트가 **특정 순간**에 의무적으로 수행하는 self-review.
> 작성 주체: 에이전트 자신 (내부 사고). 산출물이 필요한 경우 `hyperagents/logs/` 또는 `docs/reviews/`.

## 왜 프로토콜인가

페르소나 (Sage) 는 호출돼야 동작한다. 프로토콜은 **작업 흐름에 강제 포함** 되므로 놓치기 어렵다. HyperAgents 의 자기참조 원칙과 일치 — **시스템이 자신을 점검하는 루프가 내장됨**.

---

## 1. Pre-flight Review (작업 시작 전)

### 언제
- 사용자 요청이 **비자명** 하거나 **범위가 모호** 할 때
- 구현이 아닌 **결정**(아키텍처/기술 스택/UX 방향) 이 필요할 때
- 사용자가 암시적으로 "팀으로 일해라", "/effort max", 복잡한 요구 사항 제시

### 체크리스트
```
1. 가정 (Assumptions)
   - 이 요청에 어떤 가정이 깔려 있는가?
   - 그 가정이 틀릴 가능성은? (위험도: 높/중/낮)
   - 검증 방법은?

2. Unknown Unknowns
   - 도메인: 온톨로지에 없는 비즈니스 룰이 영향을 주는가?
   - 기술: 선택한 스택/라이브러리의 숨은 제약?
   - 비즈니스: 고려 안 된 이해관계자·규제·시장 변수?

3. 누락된 관점
   - 모바일 vs 데스크톱?
   - 접근성 (시각장애, 노년층)?
   - 경계 케이스 (API 실패, 빈 결과, 동시성)?

4. 프레이밍 점검 (Reframing)
   - 문제를 다르게 정의하면 더 나은 해법이 나오는가?
   - 요청을 **반대로** 뒤집어 보면? (사용자가 "X 하지 마" → "X 의 대안은 뭐?")

5. 인지 편향 (해당 시)
   - 확증 편향: 내 가설을 지지하는 증거만 모으고 있지 않은가?
   - 앵커링: 첫 제안에 과도하게 영향받고 있지 않은가?
   - 생존자 편향: 성공한 사례만 보고 실패 사례를 간과하는가?
```

### 적용 예시 (간결 모드)
단순 작업에는 장문 review 불필요. **핵심 1-2개만 빠르게 자문**:
- "이 요청의 핵심 가정은 무엇인가?"
- "내가 모르고 있을 수 있는 것은?"

### 복잡한 결정 시 산출물
`docs/reviews/pre_flight_YYYY-MM-DD_<topic>.md` 에 기록 후 사용자에게 제시:
```markdown
# Pre-flight Review: <주제>

## 요청 요약
## 가정 점검 (위험도 포함)
## Unknown Unknowns
## 누락된 관점
## 방향 제안 (필요 시)
## 추천: 진행 / 검토 후 진행 / 방향 재고
```

## 2. Checkpoint Review (마일스톤 중간 점검)

### 언제
- PRD 확정 전
- 큰 구현 완료 후 배포 전
- 여러 세션 연속 작업 중 방향성 의심 시
- 같은 실수가 2회 이상 반복

### 체크리스트
```
1. 설계 검증
   - 지금 구현한 설계가 6개월 후에도 유효한가?
   - 되돌리기 어려운 결정이 섞여 있지 않은가?

2. 테스트 공백
   - 우리가 **테스트하지 않은 시나리오** 는?
   - 실패 모드 중 관찰이 어려운 것은?

3. 진행 중 드리프트
   - 초기 목표와 현재 방향이 일치하는가?
   - 스코프 크리프가 일어나고 있지 않은가?

4. 레거시 축적
   - 이번 작업으로 **기술 부채** 가 쌓였는가?
   - `hyperagents/best_practices.md` 에 반영해야 할 교훈이 있는가?
```

### 산출물
- 간단: `hyperagents/logs/` 세션 로그 내 "Checkpoint" 섹션
- 복잡: `docs/reviews/checkpoint_YYYY-MM-DD_<topic>.md`

---

## 3. Retrospective Review (작업 종료 후)

### 언제
- 세션 종료 시점 (의무)
- 주요 기능 완성 후

### 체크리스트 (= 기존 Self-Correction Loop 확장)
```
1. 하네스 부재/불충분
   - 이번에 발생한 에러/재작업의 원인이 `hyperagents/` 에 있었을 **규칙/도구의 부재** 인가?
   - 있다면 해당 파일 업데이트

2. 반복 패턴 식별
   - 같은 해결책을 두 번 썼는가? → `best_practices.md` 후보
   - 같은 실수를 두 번 했는가? → `best_practices.md` 의 "금지 패턴" 후보

3. 사용자 피드백 추출
   - 사용자가 명시적으로 승인한 판단 → `coding_standards.md` 후보
   - 사용자가 거절한 접근 → `best_practices.md` 의 "반대 케이스"

4. 성공 요인 분석
   - 무엇이 효과적이었나? 왜?
   - 다른 상황에도 적용 가능한가?

5. 하네스 진화 기록
   - `hyperagents/evolution.md` 에 한 줄 이상
   - 새 스크립트 / 카테고리 신설 / CLAUDE.md 갱신 등 구조적 변화는 필수 기록
```

### 산출물
- `hyperagents/logs/YYYY-MM-DD-<topic>.md` (필수)
- 구조적 변경 시 `hyperagents/evolution.md` 업데이트 (필수)
- 인사이트 승격 시 `best_practices.md` / `coding_standards.md` 갱신

## 작동 원칙

### 비례성 (중요)
- 모든 작업에 세 리뷰를 **전부** 돌리지 말 것
- 단순 bug fix → Retrospective 만
- UX 중대 변경 → Pre-flight + Retrospective
- 아키텍처 결정 → 셋 다

### 자기참조
- 이 review_protocols.md 자체도 retrospective 대상
- "이 프로토콜이 실제로 도움이 되었나?" 를 세션 몇 회 후 점검

### 겸손
- "내가 틀릴 수 있지만, 이것은 점검해볼 가치가 있다" 의 자세
- 강요하지 말고 **제안** — 사용자·상위 결정을 방해하지 않음

### 비차단
- 리뷰가 작업을 멈추게 하지 않음. 발견된 리스크는 **기록만** 하고 진행 가능
- 단, "위험도 높음" 은 사용자에게 명시적으로 에스컬레이션

## 트리거 요약표

| 상황 | 적용 프로토콜 |
|---|---|
| 사용자 단순 요청 ("버튼 색 바꿔줘") | 없음 (바로 작업) |
| 중급 요청 ("새 화면 만들어줘") | Pre-flight (간결 모드) + Retrospective |
| 복잡 요청 ("시스템 재설계") | 세 리뷰 모두 + 산출물 필수 |
| 에러/재작업 반복 | Checkpoint 즉시 |
| 같은 실수 2회 | Retrospective 조기 실행 |
| 세션 종료 | Retrospective (의무) |
| "팀으로 일해라" | Pre-flight + Checkpoint + Retrospective |
